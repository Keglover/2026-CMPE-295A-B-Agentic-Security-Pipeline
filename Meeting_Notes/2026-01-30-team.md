Realistic Project Directions
Based on what's happening with Moltbot/OpenClaw right now, here are directions that leverage your team's specializations:
Direction 1: Agentic AI Security Scanner/Auditor
Build a tool that scans AI agent configurations, skills/plugins, and MCP server connections for security vulnerabilities.
Why it's relevant: Cisco just released an open-source Skill Scanner tool because 26% of 31,000 agent skills analyzed contained at least one vulnerability. Cisco Blogs There's massive need here.
Team fit:
Security specialist → vulnerability classification and detection rules
Data science → pattern detection in malicious vs. benign skills
Cloud/virtualization → containerized scanning infrastructure

Direction 2: Sandboxed Execution Environment for AI Agents
Design a secure-by-default deployment architecture for local AI agents.
Why it's relevant: Moltbot's own documentation acknowledges there is "no perfectly secure setup when operating an AI agent with shell access." Without explicit sandboxing configuration, the agent has full access to everything the user can access. Token
Team fit:
Cloud/virtualization → VM isolation, container security, firewall rules
Security → least-privilege access models, credential vaulting
Data science → behavioral anomaly detection for agent actions

Direction 3: Prompt Injection Detection Pipeline
Build a real-time detection system for prompt injection attacks targeting AI agents.
Why it's relevant: Moltbot does not maintain enforceable trust boundaries between untrusted inputs (web content, messages, third-party skills) and high-privilege reasoning or tool invocation. As a result, externally sourced content can directly influence planning and execution without policy mediation. Palo Alto Networks
Malicious links from unknown senders will be treated with the same level of security as a message from family. Attack payloads can be hidden inside a "Good morning" message forwarded on WhatsApp or Signal. Palo Alto Networks
Team fit:
Data science → ML-based detection of injection patterns
Security → attack taxonomy, red-team test cases
Cloud → scalable inference pipeline

Direction 4: Supply Chain Security for AI Agent Skills/Plugins
Create a vetting and trust framework for third-party AI agent extensions.
Why it's relevant: A security researcher published a malicious skill on the official MoltHub registry and inflated its download count, so it became the most popular asset. In less than eight hours, 16 developers in seven countries downloaded the artificially promoted skill. Bleeping Computer
Team fit:
Security → code signing, reputation systems, static analysis
Data science → popularity manipulation detection
Cloud → secure distribution infrastructure

A) “Agent Firewall” — Policy-mediated tool execution
Idea: Put a security “gate” between the LLM and tools.
LLM proposes an action → Policy Engine decides allow/deny/ask-user


Tools run only through a mediator that enforces:


allowlists (commands, domains)


redaction / DLP checks on outputs


step-up confirmation for risky actions (“Are you sure?”)


audit logs of every decision/action


Why it’s solid: maps directly to OWASP issues like prompt injection + insecure output handling + excessive agency.
Demo: show an injected web page trying to trick the agent into exfiltrating secrets → firewall blocks + logs.
B) “Secure Local Agent Runtime” — Sandbox + least privilege by default
Idea: The agent can try anything, but the environment constrains damage.
Run tools inside containers / restricted users


Filesystem jail (workspace-only)


Network egress control (domain allowlist)


Secrets via vault (scoped tokens, short-lived)


“No sandbox by default” is a known pain point in this space.


Demo: baseline agent can read ~/.ssh (or a test secrets folder); secured runtime can’t.
C) “Prompt-Injection Resistant Browsineg/RAG” — Treat content as hostile
Idea: Focus specifically on indirect prompt injection via web/RAG:
Strip/neutralize “instructions” in retrieved content


Separate “data” vs “instructions” channels


Require citations/grounding for tool-triggering claims


Enforce “reference monitor”: web content can suggest, never command


This matches what’s being discussed around agentic browsing risks.
Demo: browse a malicious page that includes hidden “system-style” instructions → agent ignores it and only extracts facts.
